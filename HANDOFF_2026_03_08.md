## What Was Done

### Plan Execution Pipeline — Full Fix (Generate → Compile → Validate → Execute)

**Root issues fixed:**

1. **Preflight display bug** (`plans.ts:parsePreflight`) — Individual checks used `c.passed` (undefined) instead of `c.status` (string "pass"/"fail"). All checks showed red X even when passing.

2. **Preflight stub** (`preflight_stubs.go`) — `StubToolContractLoader.GetToolVersionStatus` returned `"active"` instead of `"published"`, causing `tool_versions_available` check to fail.

3. **Execute 500 — workflow name collision** (`plan_generator.go`) — Real workflow executor creates workflow named `plan_{planID}`. On retry, "already exists" error. Fixed with UUID suffix for unique names.

4. **Execute 500 — node ID format** (`plan_generator.go`) — Plan compiler generates node IDs with dots (e.g., `node_000_cad.geometry_check`) which the workflow validator rejects. Added stub fallback until plan compiler is fixed.

5. **409/412 error handling** (`plans.ts`) — Frontend now handles:
   - `PLAN_ALREADY_EXISTS` (409) → returns existing plan
   - `PLAN_NOT_DRAFT` (409) on validate/compile → returns existing plan
   - `PLAN_NOT_VALIDATED` (412) on execute → auto-validates then retries

6. **Smart retry** (`PlanExecutionPanel.tsx`) — "Retry" button now retries from the failed step (not always from generate). Tracks `failedAt` state.

7. **Stub execution** (`plan_generator.go`) — When no real workflow engine is available, stub immediately transitions plan to `completed` (was stuck in `executing` forever).
